Current setup:
  - saveFile() writes to disk at uploads/filename
  - DB stores /uploads/filename (a relative path)
  - Backend serves the uploads/ directory as static files
  - Frontend prepends http://localhost:8080 to every image path — hardcoded in 8 places

  ---
  Two approaches

  Option A: Server-side upload (simpler)

  Frontend sends the file to your backend exactly as it does now. Backend uses the AWS SDK (R2 is S3-compatible) to
  stream it to R2 instead of disk. Stores the full public R2 URL in the DB.

  Pros: Almost no frontend change, backend stays in control
  Cons: Every uploaded image passes through your Go server first — wasted bandwidth and memory

  Option B: Presigned URLs (direct upload)

  Backend generates a short-lived presigned URL and returns it to the frontend. Frontend uploads the file directly to R2
   using that URL. Sends the resulting key back to backend to store.

  Pros: No image traffic through your server at all — cheaper, faster, scales better
  Cons: Requires a new endpoint, more frontend logic

  ---
  Recommendation: Option A now, with the right structure for Option B later

  For a project at this stage, Option A is the right call. The upload flow is already working and the files aren't huge.
   The key thing to get right from the start is how you store and serve the URL, since that decision affects both
  options equally.

  Specifically:
  - Store the full URL in the DB (e.g. https://pub-xxx.r2.dev/abc123.jpg) rather than a relative path
  - Remove the http://localhost:8080 prefix from every <img src> in the frontend — the URL from the DB should be used
  directly

  That means when you eventually switch to Option B you're just changing how the URL gets generated, not how it's stored
   or rendered.
